## Context

CuteGoals 2.0 要从空仓重新建立面向真实家庭长期使用的 MVP。仓库当前没有后端、前端、数据库迁移或测试实现，现有 2026-04-07 设计稿和 core-features change 却把全部任务标记为完成，并混用了公网多家庭、短信优先、跨家庭管理员与单家庭业务假设。该状态不能作为实施或验收基线。

本设计服务于四类利益相关方：负责部署和维护实例的管理员、管理家庭激励规则的家长、完成任务和兑换奖励的孩子，以及后续负责扩展学校/社区多家庭模式的开发者。

约束如下：

- 首发区域为中国大陆，产品只提供私有化部署分发包，不提供厂商 SaaS。
- MVP 每个实例恰好服务一个家庭，但数据模型不得阻断未来多家庭扩展。
- 部署方默认具备联网构建条件；不提供离线镜像归档。
- 支持 Windows 和 Linux 主机，以及 Linux amd64、arm64 容器。
- 后端沿用 Spring Boot 与 MyBatis-Plus 技术线，前端使用 React 和 TypeScript。
- 上线级 MVP 必须覆盖安全、隐私、幂等、并发、备份、恢复和可访问性，而不只覆盖演示主流程。

本 PRD 符合大型变更拆分信号，但用户已明确选择继续单一 core-features change。原因是当前仓库没有可独立发布的基础能力，认证、家庭、任务、积分、奖励和部署共同构成首个可用版本；将其拆开会让中间 change 无法独立验收。tasks 仍按依赖阶段设置门禁，未来机构多家庭能力必须作为独立 change。

## Goals / Non-Goals

**Goals:**

- 建立可实现、可测试、可追踪的单家庭私有化 MVP 规范。
- 完成任务创建、分配、提交、审核、积分发放、奖品兑换和履约的业务闭环。
- 在重试、并发和部分失败下保持积分、库存、审核和兑换数据一致。
- 提供实例初始化、账号、配置、审计、备份恢复和升级所需的运维能力。
- 提供管理员、家长和孩子三端一致但差异化的响应式体验。
- 保留 family_id 和清晰模块边界，为未来多家庭 change 提供演进点。

**Non-Goals:**

- 学校、社区、机构或单实例多家庭模式。
- 厂商托管 SaaS、控制面、许可证云校验或默认遥测。
- 社交、排行榜、积分转赠、支付、物流、第三方登录和原生移动应用。
- 强依赖短信服务；短信认证仅为部署方可选配置。
- 数据库迁移原地降级；失败恢复依赖升级前备份。

## Decisions

### D1：单实例单家庭，同时保留家庭作用域

MVP 初始化时创建且只允许存在一个有效家庭。所有领域记录继续携带 family_id，并通过统一的请求上下文和数据访问约束注入当前家庭；UI 和 API 不提供家庭选择或跨家庭查询。

替代方案是删除 family_id，等未来再迁移。该方案当前更简单，但会让多家庭扩展成为高风险全表迁移，因此不采用。保留 family_id 的额外成本很低，且不等同于提前实现多租户。

### D2：统一 SPA 与模块化单体

前端采用一个 React SPA，以 /admin、/parent、/child 路由区分三端，共享认证、API 客户端、领域类型和基础组件，并按路由拆包。后端采用 Spring Boot 3 Maven 多模块单体，按 common、auth、system、business 和 application/启动边界组织。

三个独立前端会增加构建和部署成本；微服务会引入对单家庭实例无价值的网络与运维复杂度，因此均不采用。模块化单体保留清晰边界，同时只需部署一个应用服务。

### D3：统一项目目录

代码和运维资产固定为 server/、web/、deploy/、docs/ 四个顶层目录。server/ 内部承载 Maven 多模块，不再同时使用 cutegoals-server/ 与 server/ 两套表述。deploy/ 保存 Compose、反向代理、环境模板、健康检查及 Bash/PowerShell 管理脚本。

### D4：本地联网构建的多平台 Compose

分发包提供源码、Compose 文档、build.sh 和等价 PowerShell 脚本。部署方在 Windows 或 Linux 主机上联网构建 Linux amd64 或 arm64 镜像。默认 Compose 拓扑包含反向代理、应用、MySQL 和 Redis，使用命名卷和健康检查控制启动依赖。

不分发离线镜像归档，避免版本和架构组合导致包体积与发布流程膨胀。未来若出现隔离网络需求，应通过独立 change 引入离线制品与签名策略。

### D5：实例初始化与角色分离

首次启动生成短期、一次性初始化令牌。完成初始化的首位用户同时获得 INSTANCE_ADMIN 和 PARENT 角色，但两类权限保持分离：实例管理员负责配置、账号、审计、备份恢复、版本和健康；家长负责家庭业务。系统必须阻止移除最后一位有效家长或最后一位实例管理员。

原跨家庭 admin 仪表盘、家庭搜索和家庭数量统计从 MVP 删除。首位管理员凭据遗失时，由部署方在本机执行维护命令触发一次性恢复流程，不联系厂商云服务。

### D6：家长认证与浏览器会话

家长必备认证方式为手机号加密码；短信验证码仅在部署方配置兼容服务商后启用。密码和 PIN 使用自适应哈希，日志、错误和审计中屏蔽凭据、令牌及完整手机号。

浏览器会话使用短期访问令牌和轮换刷新令牌，并通过 Secure、HttpOnly、合适 SameSite 属性的 Cookie 传递；状态修改请求启用 CSRF 防护。登出、密码变更、账号停用和可疑会话处置必须撤销刷新令牌。生产环境必须使用 HTTPS，HTTP 只用于本机开发。

### D7：家长授权的孩子设备绑定

孩子不使用可全局枚举的账号。家长先在已认证会话中创建短期绑定授权，设备绑定当前实例家庭后展示孩子档案，孩子选择头像并输入家庭内唯一 PIN。绑定令牌只授予孩子端范围，可由家长撤销。

连续五次失败会锁定“设备绑定加孩子档案”组合十五分钟，并叠加 IP/设备速率限制。该模型比“仅凭四位 PIN 全局登录”可定位且更安全，也比为孩子分配手机号或复杂用户名更适合目标用户。

### D8：任务快照、迟交和审核历史

周期规则是模板的可选属性。任务分配时固化模板名称、难度、奖励积分、截止时间等快照；后续模板修改只影响新分配。实例时区默认 Asia/Shanghai，周期生成和批量分配使用稳定发生键及幂等键防止重复。

审核主状态为 PENDING → SUBMITTED → APPROVED 或 REJECTED。REJECTED 保持可观察；孩子重新提交时回到 SUBMITTED。每次提交和审核形成不可覆盖的 attempt/history，驳回原因必填。同一提交只能成功审核一次，批准与积分入账处于同一事务。

逾期是根据截止时间计算的标记，不覆盖审核状态。家庭默认迟交策略可被单个分配覆盖；禁止迟交时返回稳定错误码，允许迟交时在审核界面明确标注。

### D9：不可变积分流水

积分流水是事实源，类型包括 EARN、SPEND、REFUND 和 ADJUST；账户余额和累计获得值是可校验投影。所有流水带唯一业务引用，余额不得为负，积分不做周期清零。

审核、兑换、退款与手工调整使用数据库事务、行锁或版本控制和唯一约束共同保证一致性。家长手工调整必须填写原因并进入审计；退款必须引用原消费且每笔兑换最多成功一次。

### D10：奖品、盲盒与历史快照

奖品价格和库存可修改，但历史兑换保存名称、图片引用、积分价格和结果快照。被历史记录引用的奖品只能停用或软删除。

盲盒项使用正整数相对权重，不要求合计一百。抽取前过滤停用或缺货项，再对剩余权重归一化；至少有一个可抽项时奖池才可展示和兑换。孩子确认前看到候选奖品与当前有效概率。库存扣减与抽取结果在同一事务中完成，防止并发超卖。

### D11：兑换履约和取消

明牌和盲盒兑换创建时原子扣积分、扣库存、写消费流水并保存快照。客户端必须提交幂等键；相同键重试返回原结果，参数冲突返回稳定错误。

兑换状态为 PENDING_FULFILLMENT → FULFILLED 或 CANCELLED。只有家长可兑现或在兑现前取消；取消时原子写退款流水并恢复库存，且只能成功一次。孩子不能自行取消盲盒结果，避免反复抽取。

### D12：隐私、安全和审计

孩子档案只要求昵称、头像和家庭内身份，不要求手机号或精确生日。家长可导出、停用和删除孩子档案；删除清除个人资料，对仍需维持账务和审计一致性的历史记录匿名化。

系统默认关闭外部遥测，不向厂商服务器发送数据。敏感操作记录操作者、时间、结果和关联对象，但不记录秘密。认证失败不得泄露账号是否存在；权限不足使用稳定错误语义，不可见资源按不存在处理以降低枚举风险。

### D13：备份、升级和非功能基线

Compose 提供每日自动备份能力，默认保留七个日备份和四个周备份，覆盖数据库与用户上传文件。恢复必须有脚本、文档和演练证据；目标 RPO 不超过二十四小时，标准恢复 RTO 不超过两小时。

数据库迁移使用版本化、前向迁移。升级前强制成功备份；升级失败时恢复旧版本镜像和备份，不承诺对已迁移数据库执行原地 down migration。

参考环境为 2 vCPU、4 GB 内存；单实例至少支持二十个并发会话和十万条业务记录。除上传和外部短信外，API P95 不超过 300 ms；局域网主要页面 LCP 不超过 2.5 秒。所有动画支持减少动态效果，滑动交互提供按钮替代。

## Risks / Trade-offs

- [保留 family_id 可能被误解为已支持多家庭] → 在配置、API 和 UI 中硬性限制单家庭，并将多家庭列为后续独立 change。
- [Windows 与两种 CPU 架构增加构建矩阵] → 应用只发布 Linux 容器，使用统一 Compose 和多架构构建脚本，在 CI 中覆盖目标矩阵。
- [HttpOnly Cookie 在纯 HTTP 局域网无法满足安全属性] → 生产文档要求配置 HTTPS；仅 localhost 开发配置允许降级。
- [家长取消兑换会改变积分和库存] → 使用原消费引用、唯一退款约束和单事务恢复，拒绝重复或已履约取消。
- [随机概率测试可能不稳定] → 单元测试使用可控随机源验证区间映射；统计测试只作为补充并声明样本与容差。
- [本地部署的可用性受部署方环境影响] → 提供健康检查、诊断命令、备份恢复演练和明确参考资源，不承诺厂商托管 SLA。
- [孩子资料删除与历史一致性冲突] → 删除个人字段并匿名化历史主体标识，不删除账务和审计所需的不可变事件。

## Migration Plan

1. 以本 change 重写 proposal、design、delta specs 和 tasks，全部任务恢复为未完成，并通过 OpenSpec strict validation。
2. 将 2026-04-07 设计文档标记为被新版重基线设计取代，保留历史链接。
3. 从空数据库以版本化迁移创建 2.0 schema，不导入已删除的 1.0 数据。
4. 实施阶段按 capability 建立单元、集成、权限、E2E、Compose 和恢复测试，再逐项勾选 tasks。
5. 首次部署通过一次性令牌初始化家庭和首位双角色用户。
6. 后续版本升级前执行备份和恢复可用性检查，迁移成功后才切换应用版本。

回滚策略：文档阶段可通过 Git 还原本 change；应用升级阶段回到上一版本镜像并恢复升级前备份。任何回滚都不得通过删除或手工改写不可变业务流水实现。

## Open Questions

当前没有阻塞 MVP 实施的开放问题。短信服务商适配、离线安装包和机构多家庭模式均明确延后，必须通过后续 change 重新设计和验收。
