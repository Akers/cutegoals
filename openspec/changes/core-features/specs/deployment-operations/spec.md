## ADDED Requirements

### Requirement: 单家庭私有化分发包
MVP SHALL 以“每个部署实例服务一个家庭”为隔离边界，并支持在中国大陆的 Windows 与 Linux 主机上通过 Docker Compose 私有化部署。分发包 MUST 提供 Compose 文件、部署文档、环境变量模板、Bash 构建脚本 `build.sh` 和 PowerShell 构建脚本 `build.ps1`（或保持同等入口的等价脚本）；默认流程 SHALL 联网获取依赖并本地构建，不要求附带离线容器镜像包。

#### Scenario: Linux 主机默认部署
- **WHEN** 运维人员在满足先决条件的 Linux 主机上复制环境模板、填写必填值并运行文档中的 Bash 构建和 Compose 启动命令
- **THEN** 系统 MUST 联网完成本地构建并启动单家庭实例，且文档中的健康验证 MUST 通过

#### Scenario: Windows 主机默认部署
- **WHEN** 运维人员在满足先决条件的 Windows 主机上使用 PowerShell 构建入口和 Docker Compose 启动命令
- **THEN** 系统 MUST 产生与 Linux 流程等价的应用版本、服务拓扑和持久化行为

#### Scenario: 默认构建时网络不可用
- **WHEN** 联网构建无法访问文档列明的镜像源或依赖源
- **THEN** 构建脚本 MUST 以非零状态退出并报告 `BUILD_DEPENDENCY_UNAVAILABLE` 及失败来源，不得宣称支持未提供的离线安装

#### Scenario: 环境模板与真实机密分离
- **WHEN** 用户检查分发包和构建产物
- **THEN** 分发包 MUST 包含仅有说明或占位值的环境模板，且 MUST 不包含任一部署实例的真实密码、令牌、私钥或家庭数据

### Requirement: 双架构 Linux 容器构建
应用容器目标平台 MUST 包含 `linux/amd64` 与 `linux/arm64`。Bash 和 PowerShell 构建入口 SHALL 接受或自动选择受支持的目标平台，并 MUST 对不受支持的平台给出明确失败；同一应用版本在两种架构上 MUST 暴露等价功能、版本和健康状态。

#### Scenario: 构建 amd64 镜像
- **WHEN** 运维人员在构建入口指定 `linux/amd64`
- **THEN** 构建流程 MUST 成功生成可由 amd64 Linux 容器运行时启动的该版本镜像

#### Scenario: 构建 arm64 镜像
- **WHEN** 运维人员在构建入口指定 `linux/arm64`
- **THEN** 构建流程 MUST 成功生成可由 arm64 Linux 容器运行时启动的该版本镜像

#### Scenario: 指定不支持的平台
- **WHEN** 运维人员指定不在支持清单中的容器平台
- **THEN** 构建脚本 MUST 在构建前以非零状态退出并报告 `UNSUPPORTED_PLATFORM` 及受支持平台列表

### Requirement: 默认 Compose 服务拓扑与持久化
默认 Compose 配置 MUST 包含反向代理、应用、MySQL 和 Redis 四类服务，并为数据库、Redis 持久数据和用户上传文件配置命名卷。每个关键服务 MUST 提供可执行健康检查；应用和反向代理的启动就绪依赖 MUST 基于下游健康状态而非仅基于容器启动顺序。默认对主机公开的业务入口 MUST 仅经过反向代理。

#### Scenario: 全新启动
- **WHEN** 运维人员使用合法环境配置在空卷上启动默认 Compose
- **THEN** MySQL 和 Redis MUST 先达到健康状态，应用 MUST 完成初始化和版本化迁移后达到健康状态，反向代理随后 MUST 提供业务入口

#### Scenario: 依赖服务未健康
- **WHEN** MySQL 或 Redis 容器已启动但健康检查失败
- **THEN** 应用 MUST 不得报告就绪，启动诊断 MUST 标识 `DEPENDENCY_UNHEALTHY`、失败服务和可执行的检查建议

#### Scenario: 重建容器保留数据
- **WHEN** 运维人员停止并重建应用、数据库、Redis 和反向代理容器但保留命名卷
- **THEN** 数据库业务记录和用户上传文件 MUST 保持可用，且不得因容器重建创建第二个家庭实例

#### Scenario: 仅代理暴露业务入口
- **WHEN** 使用默认生产 Compose 检查主机端口
- **THEN** MySQL、Redis 和应用内部端口 MUST 不向非本机网络直接公开，外部业务流量 MUST 通过反向代理

### Requirement: 参考容量与性能目标
默认 Compose 在参考最低环境 2 vCPU、4 GB 内存、10 万条业务记录和至少 20 个并发会话下 SHALL 保持核心业务可用。除文件上传、下载及外部短信调用外，核心 API 在稳定的代表性混合负载窗口内的 P95 响应时间 MUST 小于或等于 300 毫秒；性能报告 MUST 分别标识被排除的外部或大文件操作。

#### Scenario: 最低环境容量验证
- **WHEN** 在 2 vCPU、4 GB 内存的主机上加载 10 万条业务记录并维持 20 个已认证并发会话执行典型读取和写入流程
- **THEN** 系统 MUST 保持健康，且积分、库存、审核与兑换事务 MUST 继续满足一致性要求

#### Scenario: 核心 API P95 验收
- **WHEN** 参考数据与并发会话在预热后持续执行不少于 10 分钟的代表性核心 API 混合负载
- **THEN** 被测核心 API 的总体 P95 响应时间 MUST 小于或等于 300 毫秒，上传、下载和外部短信耗时 MUST 从该指标单独报告

#### Scenario: 数据量达到十万条后的分页
- **WHEN** 兑换、任务或流水历史累计达到 10 万条业务记录并查询靠后的分页
- **THEN** API MUST 返回有界分页结果，不得因一次请求加载全部历史而使应用失去健康状态

### Requirement: 生产传输与机密安全
生产模式 MUST 使用 HTTPS；明文 HTTP SHALL 仅允许绑定本机地址的开发模式，生产 HTTP 请求 MUST 重定向至 HTTPS 或被拒绝。系统 MUST 不提供跨安装通用的默认密码，生产启动 MUST 拒绝空值、示例值或已知默认机密。机密和环境值 MUST 不写入容器镜像层、客户端资源、健康响应或日志，运行时默认 MUST 禁用外部遥测。

#### Scenario: 生产环境未配置 HTTPS
- **WHEN** 生产模式缺少有效 HTTPS 配置或尝试仅以明文 HTTP 对外监听
- **THEN** 系统 MUST 拒绝进入就绪状态并报告 `TLS_REQUIRED`，不得回退为对外明文服务

#### Scenario: 本机开发使用 HTTP
- **WHEN** 开发模式将 HTTP 仅绑定到回环地址
- **THEN** 系统 SHALL 允许本机开发访问，并 MUST 在诊断中明确标识该模式不可用于生产

#### Scenario: 使用空值或示例密码启动生产
- **WHEN** MySQL、应用会话或其他必需机密为空、仍为模板占位值或等于分发包已知示例值
- **THEN** 启动 MUST 失败并报告 `CONFIG_INVALID` 和字段名称，但 MUST 不回显机密内容

#### Scenario: 检查镜像和日志中的机密
- **WHEN** 验收人员检查应用镜像层、静态前端文件、健康与版本响应以及标准运行日志
- **THEN** 这些位置 MUST 不包含环境文件内容、密码、令牌、私钥或连接串中的机密部分

#### Scenario: 默认运行时无遥测外连
- **WHEN** 使用默认生产配置运行核心家庭功能且未显式启用短信等外部集成
- **THEN** 应用 MUST 不向外部分析、崩溃收集、许可证或控制平面发送遥测，核心功能 MUST 不依赖此类外连

### Requirement: 自动备份与保留策略
部署包 MUST 提供可由 Windows 与 Linux 调度的备份脚本和说明。默认计划 SHALL 每日至少生成一次一致性备份，并 MUST 同时包含 MySQL 业务数据与用户上传文件；保留策略 MUST 保存最近 7 份每日备份及最近 4 份每周备份。每份备份 MUST 带有时间、应用版本、数据版本和完整性校验信息，失败不得删除最后一份已验证可用备份。

#### Scenario: 每日备份成功
- **WHEN** 自动备份计划到期且数据库与上传卷可访问
- **THEN** 备份流程 MUST 生成包含数据库和上传文件的一致备份、验证其完整性并以成功状态结束

#### Scenario: 执行保留清理
- **WHEN** 已存在超过 7 个每日保留点和超过 4 个每周保留点
- **THEN** 清理流程 MUST 保留最近 7 份每日备份和最近 4 份每周备份，并 MUST 在新备份验证成功后才删除超期备份

#### Scenario: 数据库备份成功但上传备份失败
- **WHEN** 数据库导出完成而上传文件复制或校验失败
- **THEN** 本次备份 MUST 标记为不可恢复并以非零状态报告 `BACKUP_FAILED`，不得将其计为最新有效恢复点

#### Scenario: 备份日志安全
- **WHEN** 备份成功或失败并输出日志
- **THEN** 日志 MUST 包含时间、阶段、备份标识和诊断码，但 MUST 不包含数据库密码、会话机密或用户文件内容

### Requirement: 恢复目标与恢复演练
部署包 MUST 提供适用于备份格式的恢复脚本和演练说明，并 SHALL 在标准恢复中校验备份、停止或隔离写流量、恢复数据库与上传文件、启动服务并验证版本和健康。默认每日备份策略 MUST 支持 RPO 不超过 24 小时；在参考最低环境且最近有效备份可访问时，标准恢复 RTO MUST 不超过 2 小时。恢复演练结果 MUST 可记录恢复点、开始与结束时间、校验结果和未解决问题。

#### Scenario: 从最近有效备份标准恢复
- **WHEN** 运维人员在参考最低环境选择最近一份验证有效的备份并运行恢复流程
- **THEN** 系统 MUST 恢复数据库和上传文件、通过迁移兼容性与健康检查，并在流程结束时报告恢复点和应用版本

#### Scenario: 验证 RPO
- **WHEN** 每日备份按计划运行且最近一次备份成功
- **THEN** 任一时刻的最近有效恢复点与当前时间差 MUST 不超过 24 小时

#### Scenario: 验证标准 RTO
- **WHEN** 从有效备份可访问且恢复先决条件满足时开始计时，直至应用健康且抽样业务记录与上传文件通过验证
- **THEN** 标准恢复 MUST 在 2 小时内完成并记录实际耗时

#### Scenario: 备份校验失败
- **WHEN** 恢复脚本发现清单缺失、校验和不匹配、数据库或上传部分不完整
- **THEN** 恢复 MUST 在覆盖现有数据前停止并报告 `RESTORE_BACKUP_INVALID`，不得把损坏备份标记为成功恢复

#### Scenario: 恢复过程失败
- **WHEN** 数据库导入、上传恢复、启动或恢复后校验任一步骤失败
- **THEN** 流程 MUST 以非零状态报告 `RESTORE_FAILED`、失败阶段和安全处置建议，且不得宣告实例健康

#### Scenario: 执行恢复演练
- **WHEN** 运维人员按文档在隔离环境执行恢复演练
- **THEN** 演练 MUST 验证数据库记录、用户上传、登录和至少一个核心业务流程，并保存 RPO、RTO 与完整性结果

### Requirement: 版本化升级与失败恢复
应用升级 SHALL 使用有序、可审计且仅执行一次的版本化数据迁移。升级流程 MUST 在更换应用版本或执行迁移前创建并验证完整备份；备份失败 MUST 中止升级。迁移或新版本健康检查失败时，流程 MUST 使用升级前备份恢复数据库和上传文件并恢复先前应用版本。系统 MUST 不承诺对已迁移数据执行原地降级，升级文档 MUST 明确失败恢复依赖备份。

#### Scenario: 正常升级
- **WHEN** 当前版本受支持、升级前备份验证成功且所有版本化迁移执行成功
- **THEN** 系统 MUST 启动目标版本、通过健康检查并记录起始版本、目标版本和已执行迁移

#### Scenario: 升级前备份失败
- **WHEN** 升级流程无法生成或验证数据库与上传文件备份
- **THEN** 升级 MUST 在迁移和替换应用前中止并报告 `PRE_UPGRADE_BACKUP_FAILED`，当前版本和数据 MUST 保持运行前状态

#### Scenario: 迁移失败后恢复
- **WHEN** 任一版本化迁移失败或目标版本无法达到健康状态
- **THEN** 升级流程 MUST 报告 `MIGRATION_FAILED` 或 `UPGRADE_HEALTHCHECK_FAILED`，并 MUST 从升级前备份恢复数据与上传文件后恢复先前应用版本

#### Scenario: 请求原地降级
- **WHEN** 运维人员尝试用旧版本直接启动已由新版本迁移的数据
- **THEN** 系统 MUST 阻止不受支持的原地降级并报告 `DOWNGRADE_NOT_SUPPORTED`，文档 MUST 指向“恢复升级前备份”的流程

### Requirement: 版本、健康与启动诊断
运行中的实例 SHALL 通过管理界面或受保护的状态接口显示应用版本、数据迁移版本和关键服务健康摘要，并 MUST 同时向 Compose 提供不含机密的机器可判定健康状态。构建、启动、备份、恢复和升级脚本 MUST 使用非零退出状态与稳定诊断码报告失败；启动失败信息 MUST 标识失败阶段、相关服务或配置字段以及至少一个安全检查建议。

#### Scenario: 查看运行版本与健康
- **WHEN** 获授权的运维人员查看实例状态
- **THEN** 系统 MUST 显示应用版本、数据迁移版本、反向代理、应用、MySQL 和 Redis 的健康摘要，且不得暴露机密

#### Scenario: Compose 判断应用未就绪
- **WHEN** 应用进程存在但迁移未完成或关键依赖不可用
- **THEN** 应用健康检查 MUST 返回未就绪，Compose MUST 不得将该实例判定为健康

#### Scenario: 缺少必填环境项启动
- **WHEN** 启动校验发现必填环境项缺失或格式无效
- **THEN** 启动 MUST 以非零状态退出并报告 `CONFIG_INVALID`、字段名称和修复说明，但不得输出该字段的机密值

#### Scenario: 启动迁移失败
- **WHEN** 应用启动期间的数据迁移失败
- **THEN** 应用 MUST 保持未就绪并报告 `MIGRATION_FAILED`、失败迁移版本和恢复指引，不得继续接受业务写入
